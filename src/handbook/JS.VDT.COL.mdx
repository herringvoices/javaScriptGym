# Intro to Collections

So far, you’ve stored values one box at a time—names, ages, flags. Real apps need to move **groups** of related values around together. That’s what **collections** are for.

* **Objects** bundle related info about **one thing** (a user, a product, a post) into key–value pages—like a **binder**.
* **Arrays** keep an **ordered list** of things (cart items, search results, leaderboard scores)—like a **numbered shelf**.

> You’ll learn (next sections):
>
> * How to **make** objects and arrays.
> * How to **read** values (dot paths & indexes).
> * How to **update** them safely (mutation, the `const` lock, adding/removing pages/items).

---

## Objects (the binder)

An object groups properties about a single thing. Keys are your tab labels; values are the info on each page.

```js
const user = {
  name: "Riley",      // key: value
  age: 19,             // commas separate properties
  isMember: true       // colon (:) instead of equals (=)
};
```

Read a value with **dot notation** (think: an **address**):

```js
console.log(user.name); // "Riley"
```

> When names get weird (spaces/starts-with-number) or your key comes from a variable, use **bracket notation**:
>
> ```js
> const field = "isMember";
> console.log(user[field]); // true
> ```

---

## Arrays (the numbered shelf)

An array keeps items in order. JavaScript counts from **0**.

```js
const cart = ["notebook", "pens", "stickers"];
console.log(cart[0]); // "notebook"
console.log(cart.length); // 3
```

You’ll use arrays for lists you loop over later (think: search results, messages, players).

---

## Why two kinds?

* Use an **object** when you’re describing **one thing** with named fields: `user.name`, `book.title`.
* Use an **array** when you have **many things** of the same shape and you care about their **order**: `books[0]`, `books[1]`.

They love working together: a `user` object with a `friends` **array**, or a `cart` **array** of `item` **objects**.
